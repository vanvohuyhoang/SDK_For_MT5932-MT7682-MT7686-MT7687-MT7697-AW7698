Overview

This section introduces the Access Layer APIs.

It shows how to use the enums, structures and functions, and API prototypes to publish a message to the mesh network, and how to reply to status requests made by nodes on the mesh network.

How to manage and reply to an access message.

  • The application calls the bt_mesh_access_model_reply() function.
  • Please refer to How to create a mesh device and add models for more information about the register access message.
  • This shows how to reply to a reliable access message when it is received. A model handle, access message, and arguments are received with a registered callback.
    void msg_generic_on_off_status(uint16_t model_handle, uint8_t presentOnOff, uint8_t targetOnOff,
    uint8_t remainingTime, bt_mesh_access_message_rx_t* msg)
    {
    bt_mesh_access_message_tx_t* reply;
    reply = malloc(sizeof(bt_mesh_access_message_tx_t));
    reply->opcode.company_id = MESH_MODEL_COMPANY_ID_NONE;
    reply->opcode.opcode = Generic_OnOff_Status;
    reply->length = 3;
    reply->buffer = malloc(3);
    reply->buffer[0] = presentOnOff;
    reply->buffer[1] = targetOnOff;
    reply->buffer[2] = remainingTime;
    bt_mesh_access_model_reply(model_handle, msg, reply);
    free(reply->buffer);
    free(reply);
    }
    void _generic_onoff_set_handler(uint16_t model_handle, bt_mesh_access_message_rx_t* msg, void* arg, bool reliable)
    {
    mesh_onoff_server_model_t* onOffServer;
    if (reliable) {
    msg_generic_on_off_status(model_handle, gOnOffServer.onOff, msg->buf[0], 0, msg);
    }
    }
    void _sig_msg_handler(uint16_t model_handle, bt_mesh_access_message_rx_t* msg, void* arg)
    {
    switch(msg->opcode.opcode) {
    case Generic_OnOff_Set: {
    _generic_onoff_set_handler(model_handle, msg, arg, true);
    break;
    }
    default:
    break;
    }
    }

How to handle model publishing.

  • The application calls the bt_mesh_access_model_publish() function.
  • This shows how to implement model publishing. There are 2 types of publishing:
  1. State change publishing
    Call bt_mesh_access_model_publish everytime the model status is changed.
    static void _publish_onoff_status(void)
    {
    bt_mesh_access_message_tx_t msg;
    // prepare model status payload
    uint8_t buffer[1];
    buffer[0] = gOn_off_server->present_on_off;
    // fill in model status opcode
    msg.opcode.opcode = BT_MESH_MODEL_GENERIC_ONOFF_STATUS;
    msg.opcode.company_id = MESH_MODEL_COMPANY_ID_NONE;
    msg.buffer = buffer;
    msg.length = 1;
    // publishing current model status
    bt_mesh_status_t ret = bt_mesh_access_model_publish(gOn_off_server->model_handle, &msg);
    }
    static void _generic_on_off_set_msg_handler(uint16_t model_handle, const bt_mesh_access_message_rx_t* msg, const void* arg, bool is_unacknowledged)
    {
    ...
    // update new state from received message
    gOn_off_server->present_on_off = msg->buffer[0];
    // after updating model state, do publishing
    _publish_onoff_status();
    ...
    }
  2. Periodic publishing
    Do publishing in publish callback registered during adding model. (Please refer to section How to create a mesh device and add models)
    void _generic_onoff_publish_callback(uint16_t model_handle, void *arg)
    {
    // handle the publish flow in this callback
    printf("GenericOnOffStatus(Publish)");
    bt_mesh_access_message_tx_t msg;
    // prepare model status payload
    uint8_t buffer[1];
    buffer[0] = gOnOffServer.present_on_off;
    // fill in model status opcode
    msg.opcode.opcode = BT_MESH_MODEL_GENERIC_ONOFF_STATUS;
    msg.opcode.company_id = MESH_MODEL_COMPANY_ID_NONE;
    msg.buffer = buffer;
    msg.length = 1;
    // publishing current model status
    bt_mesh_status_t ret = bt_mesh_access_model_publish(model_handle, &msg);
    }

bt_mesh_status_t bt_mesh_access_model_publish ( uint16_t  model_handle,
const bt_mesh_access_message_tx_t *  msg 
)

Publish an access message to the publish address of the model.

Parameters
[in]model_handleis the handle of the model.
[in]msgis the message to be sent to the peer.
Returns
BT_MESH_SUCCESS, means the message is successfully sent.
BT_MESH_ERROR_NULL, means the parameter is null.
BT_MESH_ERROR_FAIL, means the operation failed. The model handle may be invalid.
